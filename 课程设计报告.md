# 创新学分申领管理平台的设计与实现

## 课程设计报告

---

## 课程设计内容与要求

本课程设计要求设计并实现一个创新学分申领管理平台。系统需支持两类用户角色：学生可管理个人信息、申领创新学分、查看比赛与教师信息、发布求助帖子并互动回复；管理员可管理比赛与教师信息、审批学分申请、发布公告、查看数据统计与可视化图表。要求采用前后端分离的微服务架构，后端基于 Spring Boot + Spring Cloud 构建，前端使用 Vue 3 框架，数据库采用 MySQL，实现完整的认证授权、学分申领审批闭环及技术社区互动功能。

## 摘要

随着高校创新创业教育的深入推进，创新学分的管理与认定成为教务管理中的重要环节。传统的纸质申请、人工审核方式存在效率低下、数据统计困难、信息追溯不便等问题。本项目基于 Spring Boot + Spring Cloud 微服务架构与 Vue 3 前端框架，设计并实现了一套创新学分申领管理平台。平台涵盖学生模块与管理员模块两大核心功能域，实现了个人信息管理、安全中心、比赛信息管理、创新学分申领审批、技术社区互动、指导教师管理等完整业务闭环。系统采用前后端分离架构，后端微服务包括认证服务、用户服务、比赛服务、学分申领服务、社区服务和网关服务，通过 Spring Cloud Gateway 统一路由与鉴权；前端基于 Vue 3 + Element Plus 构建响应式用户界面，利用 ECharts 实现数据可视化。经测试，系统功能完整、运行稳定，能够有效提升创新学分管理的规范化与信息化水平。

**关键词**：创新学分；微服务架构；Spring Boot；Vue 3；前后端分离

## 第一章 引言

### 1.1 项目背景

在当前高等教育改革的大背景下，创新创业教育已成为高校人才培养的重要组成部分。各高校普遍设立了创新学分制度，鼓励学生通过参加学科竞赛、科研项目、创新创业实践等方式获取创新学分，并将其纳入毕业要求。然而，传统的创新学分管理方式主要依赖纸质申请表和人工审批，存在以下突出问题：

1. **效率低下**：学生需要填写纸质表格、附上获奖证明复印件，提交至学院教务办，经多级审核后才能完成认定，流程繁琐且耗时较长。
2. **数据分散**：学分记录分散在各个学院和教务部门，缺乏统一的数据管理平台，难以进行全校层面的统计与分析。
3. **信息不透明**：学生无法实时了解申请进度，也难以查询自身历史获奖记录和学分积累情况。
4. **统计困难**：教务管理人员难以快速生成各年级、各学院的获奖统计报表，数据决策支持能力不足。

基于上述问题，开发一套信息化的创新学分申领管理平台具有重要的现实意义。该平台能够实现学分申领的线上化、审批流程的自动化、数据统计的可视化，显著提升管理效率和服务质量。

### 1.2 项目目标

本项目旨在设计并实现一个功能完善的创新学分申领管理平台，具体目标如下：

1. 实现学生创新学分的在线申领与管理员审批闭环，支持获奖证明图片上传与管理。
2. 提供比赛信息发布与查看功能，方便学生及时获取竞赛资讯并直接跳转报名系统。
3. 搭建技术社区平台，支持学生发布求助信息、查看公告、互动回复，促进学术交流。
4. 实现指导教师信息的集中管理与展示，方便学生查找和联系指导教师。
5. 为管理员提供数据统计与可视化功能，包括各年级获奖统计、申领状态分布、学年学分汇总等。
6. 采用微服务架构设计后端，保证系统的可扩展性与可维护性。

### 1.3 项目意义

本项目的实施具有以下意义：

- **管理规范化**：将创新学分的申领、审批、统计流程标准化，减少人为干预，提高管理的规范性和公正性。
- **服务便捷化**：学生可通过平台随时随地提交申请、查看进度、获取资讯，大幅提升用户体验。
- **决策数据化**：通过数据统计与可视化，为教务管理部门提供决策支持，助力教育质量提升。
- **技术实践性**：项目采用当前主流的微服务架构与前端技术栈，具有较好的技术实践价值和学习参考意义。

## 第二章 需求分析

### 2.1 系统角色分析

本系统涉及三类用户角色：

1. **学生（STUDENT）**：系统的主要使用者，可以管理个人信息、查看比赛信息、申领创新学分、查看个人获奖记录、发布和回复技术求助帖子、查看指导教师信息。
2. **管理员（ADMIN）**：系统的管理者，除了具备个人信息管理功能外，还可以管理比赛信息、审批学分申请、管理技术社区帖子、管理指导教师信息、查看数据统计。
3. **指导教师（INSTRUCTOR）**：其信息由管理员维护，供学生查看和联系。

### 2.2 功能需求分析

根据任务书要求，系统功能分为学生模块和管理员模块两大部分。

#### 2.2.1 学生模块功能需求

| 功能项 | 需求描述 |
|--------|---------|
| 个人信息 | 学生可修改个人信息，包括头像、昵称、个人简介、联系方式等；手机号修改需跳转至安全中心 |
| 安全中心 | 学生可修改登录密码、绑定或更换手机号 |
| 比赛信息 | 学生可查看当前时间段内可参加的比赛信息，点击可直接跳转至对应的报名系统 |
| 个人获奖 | 学生可查看当前学年内个人的获奖情况及创新学分获得情况 |
| 创新学分 | 学生可通过上传获奖证明照片自主申领创新学分，选择所属学年，查看当前学分拥有情况 |
| 技术求助 | 学生可发布求助信息、查看他人求助信息、回复他人的求助帖子，实现技术交流与互助 |
| 指导教师 | 学生可查看全校指导教师的信息，包括简介、联系方式等，点击可查看详情 |

#### 2.2.2 管理员模块功能需求

| 功能项 | 需求描述 |
|--------|---------|
| 个人信息 | 管理员可修改个人信息，包括头像、昵称等 |
| 安全中心 | 管理员可修改密码、绑定手机号 |
| 比赛信息 | 管理员可对比赛信息进行添加、查看、修改、删除（CRUD）操作 |
| 学生信息 | 管理员可查看所有学生的获奖情况和创新学分获得情况，以图表形式展示各年级获奖统计 |
| 创新学分 | 管理员可对学生提交的学分申请进行批准或驳回操作，并可填写审批备注 |

| 指导教师 | 管理员可对指导教师信息进行添加、查看、修改、删除操作 |

### 2.3 非功能需求分析

1. **安全性**：系统需实现用户身份认证与授权，采用 JWT Token 机制；不同角色的访问权限需严格隔离；网关层实现全局鉴权过滤。
2. **可用性**：系统界面需简洁友好，操作流程直观；支持分页查询、表单验证、操作反馈提示等。
3. **可扩展性**：采用微服务架构，各服务独立部署、独立扩展，新增功能模块不影响现有服务。
4. **性能**：数据库查询需建立合理的索引；前端采用懒加载、按需渲染等优化策略。
5. **兼容性**：前端页面需适配主流浏览器（Chrome、Edge、Firefox），采用响应式布局。

#### 3.2.2 微服务模块说明

项目采用 Maven 多模块管理，共划分为 7 个模块：

1. **credit-common（公共模块）**：存放各微服务共享的实体类（User）、工具类（JwtUtils）和统一响应结果（Result），避免代码重复。

5. **credit-competition（比赛模块）**：管理比赛信息的 CRUD 操作。
6. **credit-apply（申领模块）**：管理创新学分申领记录的提交、审批、查询及数据统计功能。
7. **credit-community（社区模块）**：管理技术社区帖子的发布、查看、删除及帖子回复功能。

## 第四章 详细设计与实现

### 4.1 认证与授权模块

#### 4.1.1 JWT 工具类

系统采用 JWT（JSON Web Token）实现无状态身份认证。`JwtUtils` 工具类封装了 Token 的生成和解析逻辑，支持自定义过期时间（默认 24 小时）和 Claims 数据。核心代码如下：

```java
package cn.in12.common;

import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.security.Keys;
import javax.crypto.SecretKey;
import java.nio.charset.StandardCharsets;
import java.util.Date;
import java.util.Map;

public class JwtUtils {

    private static final String SECRET = "in12-credit-system-very-secret-key-1234567890";
    private static final long EXPIRATION = 24 * 60 * 60 * 1000; // 24小时

    private static SecretKey getSigningKey() {
        byte[] keyBytes = SECRET.getBytes(StandardCharsets.UTF_8);
        return Keys.hmacShaKeyFor(keyBytes);
    }

    /**
     * 生成Token，将 userId、username、role 等信息写入 Claims
     */
    public static String createToken(Map<String, Object> claims) {
        return Jwts.builder()
                .claims(claims)
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + EXPIRATION))
                .signWith(getSigningKey())
                .compact();
    }

    /**
     * 解析Token，获取其中存储的用户信息
     */
    public static Claims parseToken(String token) {
        return Jwts.parser()
                .verifyWith(getSigningKey())
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }
}
```

#### 4.1.2 登录认证流程

认证服务 `LoginController` 接收前端提交的用户名和密码，调用 `AuthService` 进行验证。验证通过后生成 JWT Token 返回前端，Token 中包含 userId、username、role 等用户信息。

```java
@Tag(name = "认证管理")
@RestController
@RequestMapping("/auth")
public class LoginController {

    @Autowired
    private AuthService authService;

    @Operation(summary = "登录接口")
    @PostMapping("/login")
    public Result<Map<String, Object>> login(@RequestBody Map<String, String> body) {
        try {
            String username = body.get("username");
            String password = body.get("password");
            if (username == null || password == null) {
                return Result.error("用户名和密码不能为空");
            }
            Map<String, Object> data = authService.login(username, password);
            return Result.success(data);
        } catch (Exception e) {
            return Result.error(e.getMessage());
        }
    }
}
```

#### 4.1.3 网关全局鉴权过滤器

网关模块中的 `AuthGlobalFilter` 实现了 `GlobalFilter` 和 `Ordered` 接口，对所有经过网关的请求进行拦截和 Token 校验。白名单路径（如 `/auth/login`、`/doc.html`、`/uploads/**`）直接放行，其他路径必须携带有效的 Token。校验通过后将用户信息写入请求头，传递至下游微服务。

```java
@Component
@Slf4j
public class AuthGlobalFilter implements GlobalFilter, Ordered {

    private static final List<String> EXCLUDE_PATHS = Arrays.asList(
            "/auth/login", "/doc.html", "/webjars/**",
            "/v3/api-docs/**", "/*/v3/api-docs/**", "/uploads/**"
    );

    private final AntPathMatcher pathMatcher = new AntPathMatcher();

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        String path = request.getURI().getPath();

        // 1. 白名单路径放行
        for (String excludePath : EXCLUDE_PATHS) {
            if (pathMatcher.match(excludePath, path)) {
                return chain.filter(exchange);
            }
        }

        // 2. 获取并校验Token
        String token = request.getHeaders().getFirst("Authorization");
        if (token == null || token.isEmpty()) {
            return unauthorizedResponse(exchange, "未携带Token");
        }
        if (token.startsWith("Bearer ")) {
            token = token.substring(7);
        }

        // 3. 解析Token，将用户信息注入请求头传递给下游服务
        try {
            Claims claims = JwtUtils.parseToken(token);
            ServerHttpRequest mutableRequest = request.mutate()
                    .header("userId", String.valueOf(claims.get("userId")))
                    .header("username", (String) claims.get("username"))
                    .header("role", (String) claims.get("role"))
                    .build();
            return chain.filter(exchange.mutate().request(mutableRequest).build());
        } catch (Exception e) {
            log.error("Token校验失败: {}", e.getMessage());
            return unauthorizedResponse(exchange, "Token无效或已过期");
        }
    }

    private Mono<Void> unauthorizedResponse(ServerWebExchange exchange, String message) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.UNAUTHORIZED);
        response.getHeaders().add("Content-Type", "application/json;charset=UTF-8");
        String body = "{\"code\": 401, \"message\": \"" + message + "\"}";
        return response.writeWith(Mono.just(response.bufferFactory().wrap(body.getBytes())));
    }

    @Override
    public int getOrder() { return -1; }
}
```

#### 4.1.4 前端路由守卫

前端通过 Vue Router 的全局前置守卫 `router.beforeEach` 实现页面级权限控制，确保未登录用户无法访问受保护页面，不同角色只能访问授权页面。

```javascript
// credit-web/src/router/index.js (核心逻辑)

function getHomePathByRole() {
  const userInfo = JSON.parse(localStorage.getItem('userInfo') || '{}')
  if (userInfo.role === 'ADMIN') return '/dashboard'
  if (userInfo.role === 'STUDENT') return '/my-awards'
  return '/competitions'
}

router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  const userInfo = JSON.parse(localStorage.getItem('userInfo') || '{}')

  // 未登录 → 跳转登录页
  if (to.path !== '/login' && !token) {
    next('/login')
    return
  }
  // 已登录但访问登录页 → 跳转首页
  if (to.path === '/login' && token) {
    next(getHomePathByRole())
    return
  }
  // 角色权限校验
  if (to.meta?.roles && !to.meta.roles.includes(userInfo.role)) {
    next(getHomePathByRole())
    return
  }
  next()
})
```

#### 4.8.1 请求封装

前端通过 `src/api/request.js` 封装了 Axios 请求实例。请求拦截器自动在请求头中添加 Authorization Token，响应拦截器统一处理后端返回的 `Result` 格式，业务错误时弹出提示。

```javascript
import axios from 'axios'

const request = axios.create({
  baseURL: 'http://localhost:8080',
  timeout: 15000,
})

// 请求拦截器：自动携带Token
request.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers['Authorization'] = token
  }
  return config
})

// 响应拦截器：统一处理Result格式
request.interceptors.response.use(
  (response) => {
    const data = response.data
    if (data.code !== 200) {
      ElMessage.error(data.message || '操作失败')
      return Promise.reject(new Error(data.message))
    }
    return data.data
  },
  (error) => {
    if (error.response?.status === 401) {
      localStorage.clear()
      window.location.href = '/login'
    }
    return Promise.reject(error)
  }
)
```

### 6.2 关键技术实现要点

#### 6.2.1 MyBatis-Plus 自动填充与简化查询

系统所有实体类的 `create_time` 和 `update_time` 字段通过 MyBatis-Plus 的 `@TableField(fill = FieldFill.INSERT)` 和 `@TableField(fill = FieldFill.INSERT_UPDATE)` 注解结合 `MyMetaObjectHandler` 实现自动填充。业务开发中大量使用 Lambda 查询构造器（`lambdaQuery()`），在编译期完成字段映射校验，避免硬编码字段名的维护风险。

#### 6.2.2 统一响应结果封装

`Result<T>` 泛型类统一了所有接口的返回格式（code、message、data），前端响应拦截器据此统一处理成功和失败场景，显著降低了前后端联调的沟通成本。

#### 6.2.3 微服务注册与路由

Spring Cloud Gateway 通过 `application.yml` 配置路由规则，按路径前缀将请求转发至对应微服务。各微服务通过注册中心（Nacos）进行服务发现，实现负载均衡与故障转移。

#### 6.2.4 ECharts 数据可视化与交互联动

数据统计页面使用 ECharts 库实现饼图和柱状图。饼图根据后端返回的申领状态统计数据动态渲染，并注册了 `click` 事件实现图表到页面跳转的交互联动。

#### 6.2.5 帖子回复功能全链路

帖子回复功能涉及 `PostReply` 实体 → `PostReplyMapper` → `PostReplyService` → `CommunityController` 的后端完整链路，以及前端 `community.js` API 封装 → `CommunityList.vue` 界面渲染的前端链路。回复区域的显示由 `v-if="detail.postType !== 1"` 控制，公告帖不显示任何回复相关内容。

#### 6.2.6 Vue 3 Composition API 的实战应用

前端全面采用 Vue 3 Composition API（`ref`、`reactive`、`computed`、`onMounted`、`watch` 等），代码组织更灵活，逻辑复用更方便。多个列表页面共享相同的分页逻辑和表格点击交互模式。

## 第七章 总结与展望

### 7.1 项目总结

本项目成功设计并实现了创新学分申领管理平台，完成了任务书中要求的全部功能。项目的主要成果包括：

1. **完整的微服务架构**：采用 Spring Boot + Spring Cloud 构建了包含网关、认证、用户、比赛、申领、社区六个微服务的后端系统，各服务独立运行、独立部署，具备良好的可扩展性。

2. **完善的功能实现**：覆盖了学生模块（个人信息、安全中心、比赛信息、个人获奖、创新学分、技术求助、指导教师）和管理员模块（个人信息、安全中心、比赛管理、学生数据统计、学分审批、技术支持、教师管理）的全部功能需求。

3. **友好的用户界面**：基于 Vue 3 + Element Plus 构建了美观、响应式的前端界面，支持列表展示、详情弹窗、表单编辑、分页查询、图片上传预览、数据可视化等多种交互形式。

4. **灵活的权限控制**：通过 JWT 认证、网关全局过滤、前端路由守卫三层机制，实现了从接口到页面的全链路权限控制，确保不同角色只能访问授权的功能。

5. **丰富的交互体验**：实现了整行点击查看详情、分类切换浏览、帖子回复互动、图表点击跳转、图片同步删除等多种增强交互功能，提升了用户体验。

### 7.2 技术亮点

- **微服务架构设计**：将系统拆分为多个独立服务，每个服务专注于单一业务领域，降低了系统耦合度，便于团队协作开发和后期维护扩展。
- **前后端分离**：前后端通过 RESTful API 通信，前端独立部署，后端提供服务接口，提高了开发效率和部署灵活性。
- **JWT 无状态认证**：采用 JWT Token 实现无状态身份认证，服务端无需保存会话信息，适合微服务架构下的分布式部署。
- **数据可视化**：集成 ECharts 实现申领状态分布饼图和各年级获奖统计柱状图，且饼图支持点击跳转，将数据展示与业务操作联动。
- **文件生命周期管理**：实现了文件上传与同步删除机制，避免存储空间浪费。

### 7.3 不足与展望

尽管本项目已实现了任务书要求的全部功能，但仍存在一些可以改进和扩展的方向：

1. **消息通知**：目前学生无法及时获知审批结果，未来可引入消息通知功能（如站内信、邮件通知），在审批完成时自动通知学生。
2. **数据导出**：管理员数据统计页面目前仅支持在线查看，可增加 Excel/PDF 导出功能，方便线下汇报和存档。
3. **移动端适配**：当前系统主要面向 PC 端使用，未来可增加移动端适配或开发小程序版本，方便学生随时随地使用。
4. **搜索与筛选**：可增加全文搜索功能，支持按关键词搜索帖子、比赛等信息；学分申请列表可增加多条件组合筛选。
5. **容器化部署**：可引入 Docker 容器化部署，编写 Dockerfile 和 docker-compose.yml，简化部署流程，提高环境一致性。
6. **服务治理增强**：可引入 Nacos 作为服务注册中心和配置中心，引入 Sentinel 实现限流熔断，提升系统的稳定性和可靠性。
